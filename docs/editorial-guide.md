# 편집 가이드

## 기본 방향

- 한국어로 명확하고 자연스럽게 쓴다.
- 독자가 무엇을 얻는지 도입부에서 빠르게 밝힌다.
- 단순 사용법보다 문제, 선택 이유, 트레이드오프를 함께 설명한다.
- 확실하지 않은 사실은 단정하지 않고 검증하거나 불확실성을 표시한다.

## 권장 구조

1. 해결하려는 문제 또는 글의 목표
2. 필요한 배경과 제약
3. 구현이나 해결 과정
4. 선택지와 트레이드오프
5. 검증 결과와 핵심 정리

글의 성격에 따라 구조를 줄이거나 바꿀 수 있다. 모든 글에 같은 소제목을 기계적으로 적용하지 않는다.

## 문체

- 짧고 구체적인 문장을 우선한다.
- 과장, 불필요한 감탄사, 근거 없는 최상급 표현을 피한다.
- 전문 용어는 처음 등장할 때만 필요한 만큼 설명한다.
- 제목과 소제목만 읽어도 글의 흐름이 드러나게 한다.
- 기존 글을 수정할 때는 작성자의 말투와 관점을 유지한다.

## 코드와 기술 정보

- 코드 블록에는 언어를 지정한다.
- 예제는 핵심에 필요한 부분만 포함하고, 생략한 전제는 설명한다.
- 가능하면 명령과 코드를 직접 실행해 검증한다.
- 버전, 성능, 보안, 호환성 주장은 현재의 1차 자료로 확인한다.
- 실행하지 못한 예제는 검증하지 못했다는 점을 명시한다.

## 출처와 링크

- 공식 문서, 표준, 원문, 저장소 등 1차 자료를 우선한다.
- 출처는 해당 주장을 뒷받침하는 문장 가까이에 연결한다.
- 존재 여부를 확인하지 않은 URL이나 인용문을 만들지 않는다.
- 시점에 따라 변할 수 있는 내용에는 확인 날짜나 대상 버전을 적는다.

## 발행 전 확인

- 제목과 설명이 본문을 정확히 대표하는가?
- frontmatter가 `src/content.config.ts`의 스키마와 일치하는가?
- 코드와 명령이 재현 가능하거나 검증 한계가 표시되어 있는가?
- 링크와 기술적 주장이 확인되었는가?
- 초안은 `draft: true`인가?
- `pnpm check`와 `pnpm build`가 통과하는가?
